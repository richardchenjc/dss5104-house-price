# DSS5104 — House Price Prediction with Linear Models

Continuous Assessment 1 · March 2026

## Overview

Full analysis pipeline for predicting house sale prices in King County, Washington (2014), using linear regression models with systematic feature engineering, kernel methods, and OOF stacking. The goal is to show how far linear models can be pushed while maintaining full interpretability.

**Best results (deduplicated dataset, 4,553 unique properties):**

| Model | CV / OOF MAPE | Train MAPE | Test MAPE | Gap |
|---|---|---|---|---|
| Lean Ridge (20 features) — primary | 18.79% | 18.62% | 17.65% | −0.97pp |
| Extended Ridge (35 features) | 18.76% | 18.51% | 17.62% | −0.89pp |
| RBF Kernel (Nyström) | 18.41% | 15.81% | 16.73% | +0.92pp |
| Poly Kernel (Nyström) | 18.09% | 16.63% | 16.72% | +0.09pp |
| **Stacked ensemble (Ridge + RBF + Poly)** | **18.02% †** | — | **16.54%** | — |
| XGBoost Conservative (benchmark) | — | 15.05% | 15.62% | +0.57pp |
| XGBoost Early Stopping (benchmark) | — | 11.21% | 15.28% | +4.07pp |

† Stacked CV column shows OOF MAPE — each prediction made on held-out data. A separate CV pass would require nested cross-validation.

The honest gap between our best linear model (16.54%) and the best XGBoost benchmark (15.28%) is **1.26 pp** — achieved without a single tree-based component and with full coefficient-level interpretability.

```bash
python analysis.py

# 3. Build the PDF report
python make_report.py

# 4. (Optional) Build the Word report
npm install -g docx
node make_report_docx.js
```

`analysis.py` will:
- Load, clean, and deduplicate `house_dataset.csv`
- Validate and fix renovation date anomalies (386 invalid entries)
- Engineer all features (lean 20-feature set + extended 35-feature set)
- Run the Lasso regularisation path analysis
- Train all models: lean Ridge, extended Ridge, RBF/Poly Nystroem kernels, OOF stacked ensemble
- Generate all figures to `figures/`
- Save `results.json`

`make_report.py` reads `results.json` and `figures/` to produce `house_price_report.pdf`.

## Key Findings

### 1. Data Integrity — Artificial Duplication
The raw 9,102-row dataset is an **artificial duplication** of 4,553 unique property sales. Five checks confirm this: all duplicate pairs share an identical sale date and zero price difference; duplicates appear at a fixed row offset of 4,551 (exactly half the dataset); and 9,098 rows are byte-for-byte identical. This inflated tree-based benchmarks by ~3-4 pp. All results use the deduplicated dataset.

### 2. Data Integrity — Renovation Date Anomalies
386 rows have `yr_renovated < yr_built`, a logical impossibility in 2014 data. Three patterns:
- **176 cases**: `yr_built=2012/2013`, `yr_renovated=1912/1923` — confirmed century-digit typos (impossible in a 2014 dataset)
- **184 cases**: `yr_built=2004`, `yr_renovated=2003` — likely pre-completion permit year
- **24 cases**: `yr_built=1966`, `yr_renovated=1963` — plausible prior-structure renovation

All 386 are treated as having no valid renovation record. `effective_age` falls back to `yr_built`; `recent_reno` is set to zero.

### 3. Feature Saturation at 20 Features
Lasso path analysis shows performance **saturates near 20 features**. Going from 20 to 38 features costs less than 0.2 pp MAPE with Ridge. The lean 20-feature model is the primary model.

### 4. Log-Transforming Area Features
Area features use `log(sqft)` rather than raw values, reflecting the concave size-price relationship: each additional sqft adds a smaller percentage increase in value as house size grows. The `log_sqft_living_sq` quadratic term further captures diminishing returns above ~3,000 sqft.

### 5. Location is the Dominant Signal
ZIP code and city target encodings are the strongest predictors. The `zip_x_sqft` interaction captures that the marginal value of floor area scales with neighbourhood prestige.

### 6. XGBoost Benchmark Interpretation
The default XGBoost (train=5.6%, test=15.5%, gap=+9.9pp) is severely overfit and not a valid benchmark. The honest Conservative configuration (gap=+0.57pp) is only **0.90 pp** better than our stacked linear model, achieved without any loss of interpretability.

## Feature Engineering Summary

**Lean 20-feature set (primary model):**

| Group | Features |
|---|---|
| Location (4) | `city_lp`, `zip_lp`, `zip_city_diff`, `zip_x_sqft` |
| Size (3) | `log_sqft_above`, `log_sqft_living_sq`, `log_sqft_basement` |
| Condition (2) | `condition`, `cond_x_age` |
| Age (2) | `log_house_age`, `effective_age` |
| Rooms (2) | `bathrooms`, `sqft_per_bedroom` |
| View / Waterfront (2) | `view`, `waterfront` |
| Basement (1) | `has_basement` |
| Renovation (1) | `recent_reno` |
| Interactions (2) | `size_x_condition`, `city_x_sqft` |
| Time (1) | `month_sold` |

Location encodings use **Bayesian-smoothed target encoding** (strength=30), fitted on training data only and re-fitted per CV fold.
