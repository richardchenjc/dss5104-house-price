# DSS5104 — House Price Prediction with Linear Models

Continuous Assessment 1 · March 2026

## Overview

Full analysis pipeline for predicting house sale prices in King County, Washington (2014), using linear regression models with systematic feature engineering, Lasso-based feature selection, and Nyström kernel extensions. The goal is to show how far linear models can be pushed while maintaining full coefficient-level interpretability.

**Results (deduplicated dataset, 4,553 unique properties, 80/20 split, seed=42):**

| Model | CV MAPE | Train MAPE | Test MAPE | Gap |
|---|---|---|---|---|
| **Lean Ridge (7 features) — primary** | **19.34%** | **18.69%** | **17.64%** | **−1.05pp** |
| RBF Kernel (Nyström, 7 features) | 18.53% | 16.45% | 16.88% | +0.43pp |
| Poly Kernel (Nyström, 7 features) | 18.82% | 17.32% | 16.83% | −0.49pp |
| XGBoost Conservative (benchmark) | — | 15.05% | 15.62% | +0.57pp |
| XGBoost Early Stopping (benchmark) | — | 11.21% | 15.28% | +4.07pp |

CV MAPE for Ridge is the 5-fold cross-validation MAPE at the geometric elbow (SE ±0.81pp). The entire range from n=7 to n=26 features falls within 1 SE of the CV minimum — statistically indistinguishable on this dataset.

The gap between our best kernel model (16.83%) and the honest XGBoost benchmark (15.28%) is **1.55 pp** — achieved without any tree-based component and with full interpretability.

## Reproducing Results

```bash
# 1. Install Python dependencies
pip install pandas numpy scikit-learn matplotlib seaborn reportlab xgboost

# 2. Run the full analysis pipeline (~3-5 min)
python analysis.py

# 3. Build the PDF report
python make_report.py

# 4. (Optional) Build the Word report
npm install -g docx
node make_report_docx.js
```

`analysis.py` will:
- Load, clean, and deduplicate `house_dataset.csv` (9,200 → 4,553 rows)
- Validate and fix renovation date anomalies (386 invalid entries)
- Engineer 29 candidate features across 8 groups
- Run Lasso regularisation path analysis (60 alphas, 5-fold CV with per-fold re-encoding)
- Identify the geometric elbow at n=7 features using maximum-curvature method
- Train all models: Lean Ridge (α=50), RBF Nyström kernel, Polynomial Nyström kernel
- Generate all figures to `figures/`
- Save `results.json`

`make_report.py` reads `results.json` and `figures/` to produce `house_price_report.pdf`.

## Key Findings

### 1. Data Integrity — Artificial Duplication
The raw 9,102-row dataset is an **artificial duplication** of 4,553 unique property sales. Five independent checks confirm this: all duplicate pairs share an identical sale date and zero price difference; duplicates appear at a fixed row offset of 4,551 (exactly half the dataset); and 9,098 rows are byte-for-byte identical. This inflates tree-based benchmarks by ~3–4 pp through memorisation. All results use the deduplicated dataset.

### 2. Data Integrity — Renovation Date Anomalies
386 rows have `yr_renovated < yr_built`, a logical impossibility in 2014 data. Four patterns identified:

| Pattern | Count | yr_built | yr_renovated | Treatment |
|---|---|---|---|---|
| Pre-completion permit | 184 | 2004 | 2003 | No valid renovation |
| Century-digit typo | 66 | 2012 | 1912 | Confirmed error — excluded |
| Century-digit typo | 110 | 2013 | 1923 | Confirmed error — excluded |
| Prior-structure work | 24 | 1966 | 1963 | No valid renovation |

All 386 are treated as having no valid renovation record: `effective_age` falls back to `yr_built`; `recent_reno` is set to zero.

### 3. Feature Selection — Geometric Elbow at n=7
Lasso path analysis identifies a geometric elbow (point of maximum curvature) at **7 features**. Beyond this point, every marginal CV gain is smaller than 0.5 SE (SE ≈ ±0.81pp) — statistically indistinguishable from fold-sampling noise on this dataset. The 7 selected features are:

| Feature | Group | Description |
|---|---|---|
| `city_lp` | Location | City-level mean log-price (target encoding) |
| `zip_lp` | Location | ZIP code mean log-price (target encoding) |
| `zip_x_sqft` | Location | ZIP encoding × log(sqft_living) — location scales value of size |
| `sqft_per_bedroom` | Rooms | Living area per bedroom — spaciousness and layout quality |
| `cond_x_age` | Condition | Condition × age interaction — sustained upkeep premium |
| `era_lp` | Age | Build-era mean log-price — captures U-shaped age-price curve |
| `view` | View | Ordinal view quality (0–4) |

### 4. Target Encoding — Raw Group Means, No Smoothing
City, ZIP code, and build era are target-encoded using raw group means (no regularisation toward the global mean). Empirical variance decomposition shows signal-to-noise ratio ≥ 1 for all groupings (ZIP: 1.19×, City: 0.99×), meaning group means carry more signal than noise and smoothing is not warranted. All encodings are fitted on training data only and re-fitted inside each CV fold to prevent leakage.

### 5. Location is the Dominant Signal
ZIP and city encodings are the strongest predictors. `zip_x_sqft` further captures that the marginal value of floor area scales with neighbourhood prestige — an additional 500 sqft in an expensive ZIP contributes more than the same space in a cheaper one.

### 6. XGBoost Benchmark Interpretation
The default XGBoost (train=5.6%, test=15.5%, gap=+9.9pp) is severely overfit and not a valid upper bound. The Conservative configuration (depth=4, gap=+0.57pp) is the honest comparator. Our best kernel model trails it by 1.2 pp — a gap that falls within typical CV variance.

## Feature Engineering Summary

**29 candidate features** submitted to Lasso selection, organised by group. Seven features were evaluated but excluded after collinearity audit (r > 0.92 with retained features): `size_x_condition`, `size_x_floors`, `any_view`, `view_x_wf`, `has_basement`, `city_x_sqft`, `zip_x_cond`.

| Group | Candidate features |
|---|---|
| Location (4) | `city_lp`, `zip_lp`, `zip_city_diff`, `zip_x_sqft` |
| Size (7) | `log_sqft_living`, `log_sqft_above`, `log_sqft_basement`, `log_sqft_lot`, `log_sqft_living_sq`, `living_to_lot`, `basement_ratio` |
| Rooms (4) | `bathrooms`, `floors`, `sqft_per_bedroom`, `bath_bed_ratio` |
| Condition (3) | `condition`, `cond_x_age`, `top_condition` |
| Age (5) | `era_lp`, `house_age`, `log_house_age`, `effective_age`, `is_new` |
| Renovation (3) | `was_renovated`, `recent_reno`, `reno_lag` |
| View / Waterfront (2) | `view`, `waterfront` |
| Time (1) | `month_sold` |

`era_lp` (build era encoding) is classified under **Age**, not Location: it captures the non-linear U-shaped age–price relationship non-parametrically and is not a proxy for geographical location.
